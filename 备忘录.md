## 1. 当前项目可以复用什么，必须替换什么

**可以复用的核心思想**是：把因子表达式看成 token 序列，用 Transformer 生成公式，再用 VM 执行公式并通过回测奖励训练生成器。`model_core/vocab.py` 目前把特征定义为 `RET / LIQ_SCORE / PRESSURE / FOMO / DEV / LOG_VOL`，`ops.py` 定义了 `ADD/SUB/MUL/DIV/NEG/ABS/SIGN/GATE/JUMP/DECAY/DELAY1/MAX3` 等算子，`StackVM` 负责解释 token 序列。([GitHub][3]) `engine.py` 的训练流程是批量采样公式、VM 执行、`MemeBacktest` 打分、更新策略生成器，最后保存 `best_meme_strategy.json`。([GitHub][4])

**必须替换的部分**主要是四块。第一，数据层现在围绕 token address、liquidity、fdv、Birdeye/DexScreener 设计，数据库表也是 `tokens` 和 `ohlcv`，不适合 A 股的股票代码、复权、财报、行业、市值、停复牌、指数成分等数据结构。([GitHub][5]) 第二，因子特征是 meme 币风格，例如流动性健康度、FOMO 成交量加速度、pump deviation 等，A 股应改成价量、换手、市值、估值、财务、行业中性化、资金流和事件类特征。([GitHub][6]) 第三，回测奖励现在是“信号大于 0.85 就买、按流动性和滑点扣费”的加密交易逻辑，A 股需要 IC/RankIC、多空分组、组合收益、换手、容量、行业/市值中性、T+1、涨跌停、停牌、手续费和印花税。([GitHub][7]) 第四，`execution/` 和 `strategy_manager/` 是 Solana/Jupiter 实盘交易机器人，A 股平台初期应改成研究、回测、模拟盘和券商接口抽象，而不是直接接实盘。([GitHub][8])

另外，仓库里确实有一个 A 股/ETF 实验脚本 `times.py`，但它只是独立研究脚本：硬编码了 Tushare token、`INDEX_CODE='511260.SH'`，只对单一标的做公式搜索和样本外检查，不是平台级模块。([GitHub][9]) 这个 token 已经暴露，第一步应立即废弃并改用 `.env` 或密钥管理。

## 2. 目标架构建议：从“交易机器人”重构为“因子研发平台”

建议把项目重命名和分层，例如：

```text
alphagpt_a_share/
  data/                 # A股数据接入、清洗、日历、复权、停复牌、指数成分
  universe/             # 股票池、指数成分、ST/退市/上市天数过滤
  factor_dsl/           # 因子语法、算子、AST、VM、公式合法性检查
  factor_engine/        # 因子计算、截面标准化、中性化、缓存
  mining/               # Transformer/GP/LLM 生成因子、采样、奖励训练
  evaluation/           # IC、RankIC、分组收益、换手、容量、相关性、稳定性
  backtest/             # A股撮合、T+1、涨跌停、停牌、费用、组合构建
  factor_store/         # 因子库、版本、血缘、去重、相关性、审批状态
  research_agent/       # 自主研发 Agent：生成假设、跑实验、写报告、复核
  api/                  # FastAPI 服务
  dashboard/            # 因子看板、实验看板、回测报告、组合监控
  configs/              # yaml 配置
  tests/                # 单元测试、数据防泄漏测试、算子 golden tests
```

核心原则是：**保留 AlphaGPT 的“公式生成器”，但让它只负责生成候选因子；平台的可信度来自数据治理、无未来函数、严格样本外验证、因子库治理和可复现回测。**

## 3. 数据层：这是最优先重构点

现有 `data_pipeline` 应该完全 A 股化。建议把 `tokens` 改成 `securities`，把 `ohlcv` 改成 `bars_daily` 或 `bars_1min`，新增 `calendar`、`adj_factor`、`daily_basic`、`financials`、`index_members`、`industry`、`suspend`、`limit_price`、`st_status` 等表。Tushare Pro 官方文档列出了 A 股日线 `daily`、复权行情 `pro_bar`、每日指标 `daily_basic`、龙虎榜、质押等数据接口；AKShare 官方文档也覆盖 A 股行情、指数、财务等数据，适合作为开放数据补充；Qlib 则内置了面向中国市场的 Alpha158/Alpha360 数据集思路，可作为基准体系参考。([tushare.pro][10])

建议的数据表最少包含：

```sql
securities(
  ts_code, symbol, name, exchange, board,
  list_date, delist_date, is_st, industry_code
)

trade_calendar(
  trade_date, is_open, prev_trade_date, next_trade_date
)

bars_daily(
  trade_date, ts_code,
  open, high, low, close, pre_close,
  volume, amount,
  adj_factor,
  limit_up, limit_down,
  is_suspended
)

daily_basic(
  trade_date, ts_code,
  turnover_rate, volume_ratio,
  pe_ttm, pb, ps_ttm, total_mv, circ_mv
)

financials(
  report_period, announce_date, ts_code,
  roe, roa, gross_margin, net_profit_yoy,
  revenue_yoy, debt_to_asset, operating_cf
)

index_members(
  trade_date, index_code, ts_code, weight
)
```

这里最容易犯错的是**未来函数**。财务数据不能按报告期直接使用，必须按 `announce_date` 生效；指数成分、ST 状态、停牌状态也必须使用当日可知数据；复权价格要明确“前复权/后复权/真实成交价”的用途。`times.py` 目前只是取单标的行情、直接生成特征并做 open-to-open 收益，不足以支撑全市场横截面因子研发。([GitHub][9])

## 4. 因子 DSL：从 meme 币特征改成 A 股可解释公式语言

现有 `FEATURE_NAMES` 只有 6 个主流程特征，且高度加密货币化。([GitHub][3]) 建议把 DSL 分成三层：

**第一层：原子特征。**
价量类：`ret_1d`、`ret_5d`、`turnover`、`amount`、`volume_ratio`、`amplitude`、`gap`、`intraday_ret`。
估值类：`pb`、`pe_ttm`、`ps_ttm`、`total_mv`、`circ_mv`、`dividend_yield`。
质量/成长类：`roe`、`gross_margin`、`revenue_yoy`、`profit_yoy`、`asset_turnover`、`accruals`。
风险类：`vol_20d`、`beta_252d`、`residual_vol`、`downside_vol`。
市场状态类：`limit_up_flag`、`limit_down_flag`、`suspend_flag`、`st_flag`、`listed_days`。

**第二层：时间序列算子。**
`delay(x,n)`、`delta(x,n)`、`ts_mean(x,n)`、`ts_std(x,n)`、`ts_rank(x,n)`、`ts_zscore(x,n)`、`ts_corr(x,y,n)`、`ts_decay_linear(x,n)`、`ts_argmax(x,n)`、`ts_min/max(x,n)`。`times.py` 里已经有 `DELTA5/MA20/STD20/TS_RANK20` 的雏形，但要改成通用参数化算子，而不是硬编码几个窗口。([GitHub][9])

**第三层：横截面算子。**
这是 A 股因子平台必须补上的：`cs_rank(x)`、`cs_zscore(x)`、`winsorize(x)`、`neutralize(x, industry, log_mktcap)`、`group_rank(x, industry)`、`orthogonalize(x, existing_factors)`。没有横截面排序、中性化和去极值，自动生成的因子很容易只是市值、行业或流动性暴露。

主流程还应把 `StackVM` 升级为 **AST + 类型系统**。现在主 `engine.py` 直接采样 token，非法公式会在 `StackVM.execute()` 里返回 `None`，这会浪费大量训练样本；`times.py` 反而有更好的 strict action mask，可以借鉴到主工程里。([GitHub][11]) 建议每个 token 带上：

```python
@dataclass
class OpSpec:
    name: str
    arity: int
    input_types: tuple[str, ...]   # ts, cs, scalar, bool
    output_type: str
    min_lookback: int
    func: Callable
```

这样生成器只能生成合法公式，也可以自动计算 lookback，避免未来函数和窗口不足。

## 5. 评价与奖励函数：从“交易收益”改成“因子有效性 + 可交易性”

现有 `MemeBacktest.evaluate()` 用 sigmoid 后的绝对阈值触发交易，再按流动性惩罚滑点，输出一个中位数 fitness。([GitHub][7]) A 股因子研发平台不应该一开始就用单一交易收益当奖励，否则会严重过拟合。建议奖励函数改成多目标：

```text
reward =
  0.35 * RankIC_IR
+ 0.20 * TopBottom_Excess_Sharpe
+ 0.15 * IC_Decay_Stability
+ 0.10 * Monotonicity
- 0.10 * Turnover_Penalty
- 0.10 * Correlation_With_Existing_Factors
- 0.05 * Complexity_Penalty
- 0.05 * Capacity_Penalty
```

每个候选因子至少输出这些报告项：覆盖率、缺失率、极值比例、IC 均值、RankIC 均值、ICIR、t 值、分年度 IC、分行业 IC、分市值桶 IC、分组收益、Top-Bottom 收益、换手率、容量估计、与已有因子的相关性、样本外表现。Qlib 的 benchmark 体系也强调 alpha 与未来收益相关性、基于 alpha 构建组合后的总回报评估，这个思路可以作为评价层参考。([GitHub][12])

数据切分必须改成：

```text
train:      因子搜索/生成器训练
valid:      选择公式、调参、去重
test:       严格样本外，只能最终查看
live paper: 模拟盘观察衰减
```

不要在全历史数据上直接选最优公式。当前 `engine.py` 主流程没有严格 train/valid/test 分离；`times.py` 有一个 80/20 样本外检查雏形，但仍需扩展到全市场、多股票、多时间段。([GitHub][4])

## 6. 回测层：必须内置 A 股市场规则

A 股回测不能用加密货币那套连续交易假设。平台应实现可配置的市场约束：T+1、涨跌停、停复牌、ST/退市、上市未满 N 日过滤、买入整手/最小申报单位、成交量参与率、冲击成本、佣金、过户费、印花税。财政部、税务总局公告显示证券交易印花税自 2023 年 8 月 28 日起减半征收，券商公告中明确由出让方单边 0.1% 调整为 0.05%。([Shanghai Tax Authority][13])

特别注意：上交所已发布《上海证券交易所交易规则（2026年修订）》，发布日期为 2026 年 4 月 24 日，生效日期为 2026 年 7 月 6 日；因此交易规则不要写死，要做成按日期生效的规则配置。([SSE][14]) 回测引擎可以这样分层：

```text
SignalEngine      生成每日股票打分
PortfolioBuilder  按 TopN/行业中性/指数增强/风险预算构建目标仓位
ExecutionSimulator
  - next_open / next_vwap / close auction
  - T+1 sell constraint
  - limit-up buy fail / limit-down sell fail
  - suspension no trade
  - volume participation cap
CostModel
  - commission
  - transfer fee
  - stamp duty
  - slippage / market impact
RiskModel
  - industry exposure
  - size exposure
  - beta exposure
  - max weight / turnover / drawdown
```

Backtrader 官方文档也提醒，回测无法保证真实市场成交，滑点在真实市场中会发生；所以即使用第三方回测框架，也要把 A 股成交约束和滑点模型单独实现。([Backtrader][15])

## 7. 因子库与实验管理：这是“自主研发平台”的核心

平台不应只保存 `best_meme_strategy.json`。当前 `strategy_manager` 依赖这个 JSON 文件加载策略，组合状态也只是本地 `portfolio_state.json`。([GitHub][16]) A 股因子平台应建立正式的 factor store：

```sql
factors(
  factor_id,
  formula_hash,
  formula_text,
  token_sequence,
  feature_version,
  operator_version,
  lookback_days,
  complexity,
  created_at,
  created_by,          -- model / human / llm-agent
  status               -- candidate / approved / deprecated
)

factor_metrics(
  factor_id,
  train_start, train_end,
  valid_start, valid_end,
  test_start, test_end,
  rank_ic_mean,
  rank_ic_ir,
  top_bottom_sharpe,
  turnover,
  max_drawdown,
  capacity,
  corr_to_existing,
  report_path
)

factor_values(
  trade_date,
  ts_code,
  factor_id,
  value
)
```

每个因子进入库前都要过五道门：合法性检查、无未来函数检查、样本外检查、相关性去重、可交易性检查。这样平台才是“自主研发平台”，而不是“随机搜公式脚本”。

## 8. LLM/Agent 层：把 AlphaGPT 从“自动公式生成”升级为“自主研发工作流”

这个仓库名字叫 AlphaGPT，但主流程本质是小型 Transformer 公式生成器，不是面向研究流程的 LLM Agent。([GitHub][17]) 如果你的目标是“自主研发平台”，建议新增四类 Agent：

```text
FactorIdeationAgent
  输入：已有因子库、研报摘要、失败案例、市场状态
  输出：候选因子假设 + DSL 公式草案

FormulaCompilerAgent
  检查：字段是否存在、语法是否合法、是否未来函数、窗口是否足够

ExperimentAgent
  自动跑：IC 分析、分组回测、样本外、分年度、分行业、相关性

ReviewAgent
  输出：通过/拒绝原因、风险暴露、过拟合迹象、是否入库
```

这里要坚持“LLM 负责提出假设和写报告，数值结果必须由确定性回测系统产生”。不要让 LLM 直接判断因子有效性。

## 9. 执行层：先删 Solana，再抽象券商接口

`execution/config.py` 读取 `SOLANA_PRIVATE_KEY`，`jupiter.py` 调 Jupiter quote/swap，`trader.py` 执行 SOL 与 token 的买卖，这些都应从 A 股研究平台主干中移除或隔离。([GitHub][18]) 建议改成：

```text
execution/
  broker_base.py       # Broker 接口
  paper_broker.py      # 模拟盘
  file_broker.py       # 生成交易指令 CSV
  qmt_broker.py        # 可选，后续接券商
  risk_gate.py         # 事前风控
```

初期不要急着实盘。A 股因子平台的成熟路径应该是：研究回测 → 模拟盘 → 小资金验证 → 才考虑自动下单。平台层面要能输出目标持仓、交易清单、失败成交原因和风控拦截原因。

## 10. 工程化优化清单

第一，**配置与密钥治理**。`times.py` 里硬编码了 Tushare token，必须删除并轮换；所有 API token、数据库密码、券商账号都放 `.env`、Vault 或 KMS，仓库只保留 `.env.example`。([GitHub][9])

第二，**包结构重构**。当前很多文件是一行压缩代码，不利于维护、审查和测试；应改成标准 Python 包，增加 `pyproject.toml`、类型标注、ruff/black/mypy、pytest、pre-commit 和 CI。

第三，**可复现研究**。所有实验必须记录数据版本、股票池版本、特征版本、公式版本、训练参数、随机种子、样本区间和代码 commit。建议接入 MLflow、Weights & Biases 或自建 experiment table。

第四，**性能优化**。A 股全市场日频约数千只股票、多年数据、多因子批量计算，建议用 Parquet + DuckDB/Polars 做离线研究，用 PyTorch/Tensor 实现批量 rolling 计算；数据库负责元数据和结果索引，不要把所有大矩阵都塞进 PostgreSQL。

第五，**测试体系**。必须增加 golden tests：`delay` 不能用未来值、`ts_mean` 窗口对齐正确、财报按公告日生效、停牌日不可交易、涨停买不进、跌停卖不出、T+1 不能当天卖、复权收益计算正确。

第六，**许可证合规**。仓库是 Apache-2.0 许可证，允许修改和再分发，但你需要保留许可证、版权和修改说明。([GitHub][19])

## 12. 推荐落地顺序

第一阶段先做 **A 股研究 MVP**：完成数据接入、A 股 schema、全市场日频行情、复权、交易日历、股票池过滤、基础价量因子和 IC/RankIC 评价。不要碰实盘。

第二阶段做 **DSL/VM 重构**：把现有 token 公式生成迁移到 AST，加入合法公式 action mask、A 股算子、横截面算子、复杂度控制和去未来函数检查。

第三阶段做 **自动因子挖掘**：把 `AlphaEngine` 的 reward 改成 RankIC/分组收益/稳定性/换手/相关性/复杂度的多目标奖励，并建立 factor store。

第四阶段做 **组合回测与模拟盘**：加入 A 股市场规则、调仓、成本、涨跌停、停牌、T+1、容量和风险暴露控制。

第五阶段再做 **Agent 化研发**：让 LLM 负责提出假设、生成 DSL、调度实验、总结报告和复核失败原因，但数值判断必须由回测引擎完成。

我建议你把这次重构定位成：**AlphaGPT-A：A 股自动因子发现与因子库平台**，而不是“把 AlphaGPT 改成 A 股交易机器人”。先把因子研发闭环做扎实，后续接模拟盘和券商实盘才有意义。
